## 🤖 Assistant

很好，这套技术栈适合直接做“**第二阶段：让小镇动起来**”。建议定位为：

> **模块化单体优先 + 内部 gRPC 边界 + RabbitMQ 事件驱动 + WebSocket 实时推送**

也就是：先不拆成多个微服务，但按未来微服务的边界来写模块，避免过早复杂化。

---

## 一、第二阶段目标定义

第二阶段核心不是“更聪明”，而是“**活起来**”。

需要实现：

- 多个 NPC
- NPC 有基础人设和状态
- 小镇有时间流动
- NPC 可按日程行动
- 用户可与 NPC 对话
- NPC 行为通过 WebSocket 实时推送
- 事件进入 RabbitMQ 异步处理
- 重要事件持久化到 PostgreSQL
- 短期状态放 Redis
- 长期记忆 / 世界知识进入 Qdrant
- Agent 通过 Eino + helloAgent_go + LLM API 生成回复和行动

---

## 二、第二阶段最小可运行范围

### 必做能力

| 能力 | 说明 |
|---|---|
| 用户连接 | WebSocket 建立连接 |
| 小镇时间 | Scheduler 周期产生 `town.tick` |
| NPC 状态 | NPC 当前地点、状态、目标 |
| NPC 日程 | 根据时间触发行动 |
| 玩家对话 | 玩家向 NPC 发送消息 |
| NPC 回复 | Agent 生成回复 |
| NPC 移动 | NPC 可切换地点 |
| 事件总线 | RabbitMQ 承载异步事件 |
| 实时广播 | WebSocket 推送 NPC 行为 |
| 记忆写入 | 对话和关键事件进入 Redis / Qdrant |
| 事件日志 | PostgreSQL 持久化事件 |

### 暂缓能力

第二阶段先不要做：

- 多 Agent 复杂协作
- 复杂任务系统
- 强剧情编辑器
- 完整后台管理
- Kubernetes
- Temporal
- 复杂权限系统
- 多模型路由
- 自动剧情生成大系统

## 四、模块边界设计

建议后端内部按这些模块拆：

```text
internal/
├── app/                 # 应用启动、依赖注入
├── config/              # Viper 配置
├── logger/              # slog 初始化
├── gateway/
│   ├── websocket/       # WebSocket 网关
│   └── grpc/            # gRPC 服务入口
├── event/               # 事件模型、发布、消费
├── scheduler/           # 小镇时间 tick
├── town/                # 小镇、地点、世界状态
├── npc/                 # NPC 状态、日程、行为规则
├── agent/               # Eino + helloAgent_go 编排
├── memory/              # Redis + Qdrant + PostgreSQL 记忆
├── chat/                # 用户-NPC 对话
├── broadcast/           # WebSocket 广播
├── repository/          # GORM 数据访问
└── worker/              # RabbitMQ 消费 worker
```

### 为什么这样拆

- `gateway` 只负责接入，不写业务。
- `event` 是系统核心协议，先定义好事件类型。
- `town` 管小镇世界状态。
- `npc` 管 NPC 自身状态和日程。
- `agent` 只负责“思考与决策”。
- `memory` 管记忆，不让 Agent 直接碰数据库细节。
- `worker` 负责把事件串起来。

## 六、第二阶段运行主流程

### 1. 小镇自动运行流程

```mermaid
sequenceDiagram
    participant Scheduler as Scheduler
    participant MQ as RabbitMQ
    participant Worker as Event Worker
    participant NPC as NPC Service
    participant Town as Town Service
    participant PG as PostgreSQL
    participant WS as WebSocket Gateway
    participant Client as Client

    Scheduler->>MQ: 发布 town.tick
    MQ->>Worker: 消费 town.tick
    Worker->>NPC: 查询到期 NPC 日程
    NPC->>PG: 读取 NPC 日程和状态
    PG-->>NPC: 返回日程
    NPC-->>Worker: 返回需要行动的 NPC

    Worker->>MQ: 发布 npc.move.requested
    MQ->>Worker: 消费 npc.move.requested

    Worker->>Town: 更新 NPC 位置
    Town->>PG: 持久化 NPC 位置
    Town-->>Worker: 返回移动结果

    Worker->>MQ: 发布 npc.moved
    Worker->>WS: 广播 NPC 移动
    WS-->>Client: NPC 从 A 移动到 B
```

---

### 2. 用户对话流程

```mermaid
sequenceDiagram
    participant Client as Client
    participant WS as WebSocket Gateway
    participant MQ as RabbitMQ
    participant Worker as Event Worker
    participant Agent as Agent Service
    participant Memory as Memory Service
    participant Redis as Redis
    participant Qdrant as Qdrant
    participant LLM as LLM API
    participant PG as PostgreSQL

    Client->>WS: 发送 user.message
    WS->>MQ: 发布 user.message.sent
    MQ->>Worker: 消费 user.message.sent

    Worker->>Agent: 请求 NPC 回复
    Agent->>Memory: 获取短期上下文
    Memory->>Redis: 读取最近对话
    Redis-->>Memory: 返回短期记忆

    Agent->>Qdrant: 检索长期记忆/世界知识
    Qdrant-->>Agent: 返回相关内容

    Agent->>LLM: 生成 NPC 回复和动作
    LLM-->>Agent: 返回 reply + action

    Agent->>Memory: 保存短期记忆
    Memory->>Redis: 写入最近对话
    Memory->>Qdrant: 写入重要记忆向量
    Memory->>PG: 写入对话事件日志

    Worker->>WS: 推送 npc.replied
    WS-->>Client: 展示 NPC 回复
```

## 七、gRPC 怎么用

第二阶段建议：

> 外部主要用 WebSocket，内部模块接口用 gRPC 风格定义。

如果你当前是模块化单体，可以先定义 `.proto`，但不一定马上部署成多个进程。

## 九、Redis Key 设计

```text
ws:session:{user_id}                  # 用户连接信息
town:{town_id}:state                  # 小镇当前状态缓存
npc:{npc_id}:state                    # NPC 当前状态缓存
npc:{npc_id}:recent_messages          # 最近对话列表
npc:{npc_id}:thinking_lock            # 防止同一 NPC 并发思考
scheduler:town:{town_id}:last_tick    # 上次 tick 时间
```

建议：

- 短期对话用 `List`
- 当前状态用 `Hash`
- 锁用 `SET NX EX`
- 所有 key 加 TTL，避免无限膨胀

## 十二、Scheduler 编排

第二阶段不要复杂化，Go 内部 ticker 足够：

```text
每 5 秒真实时间 = 小镇 10 分钟
```

流程：

```mermaid
flowchart TD
    Start[Scheduler Start] --> LoadTown[加载小镇时间]
    LoadTown --> Tick[每 N 秒触发]
    Tick --> Advance[推进小镇时间]
    Advance --> Save[保存 current_day/current_minute]
    Save --> Publish[发布 town.tick]
    Publish --> Check[Worker 检查 NPC 日程]
    Check --> Event[生成 NPC 行动事件]
```

## 十三、Agent 编排策略

第二阶段 Agent 不要太复杂，采用：

> Observe → Retrieve → Think → Act → Remember

```mermaid
flowchart LR
    Observe[Observe<br/>当前事件/NPC状态] --> Retrieve[Retrieve<br/>短期记忆+Qdrant检索]
    Retrieve --> Think[Think<br/>Eino + LLM API]
    Think --> Act[Act<br/>回复/移动/状态变化]
    Act --> Remember[Remember<br/>保存对话和关键事件]
    Remember --> Publish[Publish<br/>发布 npc.replied/npc.moved]
```

### Prompt 输入建议

Agent 输入上下文包括：

```text
NPC 人设
NPC 当前状态
NPC 当前地点
最近 5-10 条对话
相关长期记忆
相关世界知识
当前小镇时间
用户输入
可执行动作列表
```

输出建议使用 JSON：

```json
{
  "reply": "当然，我刚从钟楼那边回来。",
  "emotion": "calm",
  "action": {
    "type": "stay",
    "target_location_id": null
  },
  "memory_importance": 0.6
}
```

## 十七、第二阶段验收标准

你可以用这些标准判断是否完成：

- `docker compose up` 后所有服务启动
- 系统自动产生 `town.tick`
- 至少 3 个 NPC 根据日程自动移动
- 前端 WebSocket 能收到 NPC 移动事件
- 用户可以给任意 NPC 发消息
- NPC 能基于人设回复
- 最近对话写入 Redis
- 重要记忆写入 Qdrant
- 事件日志写入 PostgreSQL
- RabbitMQ Management 能看到事件流
- 重启服务后 NPC 基础状态可恢复

## 十八、最终建议

第二阶段就按这条主线做：

> **Scheduler 驱动时间 → RabbitMQ 分发事件 → Worker 更新 NPC/小镇状态 → Agent 生成行为 → WebSocket 广播变化 → Redis/Qdrant/PostgreSQL 保存状态和记忆**

技术栈定版：

```text
Go
gRPC
WebSocket
PostgreSQL + GORM
Redis
RabbitMQ
Qdrant
Eino
helloAgent_go
LLM API
Viper
log/slog
Docker Compose
```

先做“3 个 NPC 会按日程移动 + 能对话 + 有短期记忆 + WebSocket 实时展示”，这就是一个真正可演示的 AI 小镇第二阶段 MVP。
